manga-tracker · 设计文档

v0.1 Phase 1 Kickoff · 中文漫画追踪 + 换源工具 · Repo: YeomanYe/manga-tracker

manga-tracker 是一套中文漫画追踪 + 换源工具，由两端组成：

  • 浏览器扩展（PC，Chrome/Firefox MV3）：用户在已适配的中文漫画站浏览时，自动识别作品和章节，写入本地书架；通过注入到页面的 Sync Bar 提供主操作入口
  • 移动端 Android app（Capacitor + WebView 套壳）：本地书架 + 跳转源站继续读 + 跨源切换

两端共享一份源站适配规则（JSON schema），整个工程在同一个 monorepo 内。

第一阶段（6-8 周）目标：扩展 + Android app 跑通端到端 — 识别 → 追踪 → 书架 → 跳读 → 换源；数据全部本地，不接云端；适配 3-5 个中文漫画站；不预置任何盗版源 URL，规则全靠用户运行时导入。

用户与场景

目标用户

  • 主要：18-35 岁中文漫画 / 动漫爱好者，长期在多个第三方漫画站之间跳转，需要统一书架与进度，不愿手动打卡
  • 次要：bgm.tv / AniList 用户中追漫画为主、嫌手动打卡太烦的子群

现状替代方案

方案覆盖痛点
Mihon（Tachiyomi 分叉）Android 原生阅读器，源生态成熟PC 浏览器无追踪
MAL-Sync浏览器扩展，自动写回 AniList/MAL中文站点几乎没覆盖
bgm.tv追番记录平台纯手动打卡，UI 老旧

核心使用场景

  1. PC 上：用户在已适配漫画站打开作品页 → 扩展识别 → Sync Bar 显示「+ 加入书架」→ 用户读章节 → 自动追踪进度
  2. 手机上：打开 app → 看到本地书架 → 点作品 → WebView 跳到源站章节继续读
  3. 跨源切换：当前源打不开（被封 / 维护）→ 用户手动切到另一个源继续读

关键架构原则

  1. 业务逻辑必须放共享 packages，不允许在两 apps 各写一份
  2. 依赖方向单向：apps/* → packages/*；packages/* 之间允许互依但禁止环
  3. 平台细节不外泄：chrome.* 不进 packages；Capacitor 插件不进 packages
  4. 存储统一抽象：所有持久化通过 @manga/storage-adapter
  5. 类型唯一来源：领域类型一律放 @manga/types

模块设计

@manga/types

领域类型唯一来源。文件按领域切分；类型 + Zod schema 同文件维护，schema 在下，类型 z.infer 出来。

packages/types/src/
├── manga.ts          # Manga / Author / Tag
├── chapter.ts        # Chapter / ReadingProgress
├── source-rule.ts    # SourceRule / SourceMatcher / SiteAdapter
├── bookshelf.ts      # BookshelfEntry / SortKey / FilterKey
└── index.ts          # 统一 re-export

@manga/source-rules

源规则的 wire format 多源兼容设计：直接 import 现有开源漫画工具的规则资源（legado / Mihon / 自有 v1），全部归一化到内部模型。详见 源规则 Schema 章节。

packages/source-rules/
├── src/
│   ├── schema.ts                # 内部统一模型的 Zod schema
│   ├── loader.ts                # 入口：detect format → 路由到 adapter → 归一化
│   ├── adapters/                # 各 wire format 转换器
│   │   ├── manga-tracker-v1.ts  # 我们自己的原生格式
│   │   ├── legado-v3.ts         # 阅读 App 漫画书源 import（V1 主力）
│   │   └── mihon-port.ts        # Mihon Kotlin 手动 port 后的 JSON（V2）
│   ├── runtime/                 # 选择器执行：CSS / XPath / regex / @JS
│   └── builtin/                 # 占位规则（仅 example.com）
└── rules-format.md              # 内部 schema + 各 adapter 输入说明

@manga/injector

content script 抽象。扩展和移动端 WebView 共用同一份脚本。Sync Bar UI 用 Shadow DOM 完全隔离，不污染原站样式。

packages/injector/src/
├── detect.ts         # 用 SourceRule 匹配当前 URL → 返回作品/章节信息
├── track.ts          # 章节追踪：监听 URL 变化、滚动、阅读时长
├── sync-bar/         # Sync Bar UI（Shadow DOM 隔离）
│   ├── index.ts      # 独立 React root
│   └── styles.css
└── api.ts            # 暴露给宿主的统一 API（依赖注入）

关键设计：injector 不直接调存储，通过宿主传入回调。这让 injector 完全平台无关。

// packages/injector/src/api.ts
export type SyncBarHost = {
  onAddBook: (m: Manga) => void;
  onUpdateProgress: (id: string, ch: number) => void;
  isInBookshelf: (id: string) => Promise<boolean>;
};
export function createSyncBar(host: SyncBarHost): SyncBarInstance;

@manga/storage-adapter

本地存储抽象层。两种实现共享同一接口，用同一组合约测试。

packages/storage-adapter/src/
├── adapter.ts          # 接口：BookshelfStore, ProgressStore, SettingsStore
├── indexeddb/          # 扩展用，基于 Dexie
├── sqlite/             # mobile 用，基于 capacitor-community/sqlite
└── index.ts            # 工厂：根据环境返回对应实现

源规则 Schema

核心策略：不重复造轮子。借鉴现有开源漫画工具的成熟规则资源，loader 多 format 兼容。

生态参考与兼容矩阵

项目形态中文站覆盖我们的策略
legado（阅读）漫画书源JSON + 内嵌 JS极强，中文社区主力V1 主力路径：wire format 兼容，loader 直接 import
Tachiyomi / MihonKotlin extension（JVM）强但偏英日内部模型抄它的 manga-native 接口设计；V2 经 Suwayomi-Server 桥接
MAL-Sync ChibiScriptJSON 操作链弱仅参考"规则可热更新"思路

Wire Format 判别（loader 入口）

所有传入规则 JSON 必须带 format 字段，loader 据此路由到对应 adapter，归一化到内部模型。无 format 字段 → 报错，不猜测。

{ "format": "manga-tracker-v1", "id": "...",            "rules": { ... } }
{ "format": "legado-v3",        "bookSourceUrl": "...", "ruleManga": { ... }, "ruleChapter": { ... } }
{ "format": "mihon-port",       "lang": "zh", "id": 12345, "name": "...", "selectors": { ... } }

内部模型（manga-tracker-v1 原生格式）

{
  "format": "manga-tracker-v1",
  "$schema": "https://manga-tracker.example/schema/source-rule-v1.json",
  "id": "site-id-string",
  "name": "站点显示名",
  "version": "1.0.0",
  "matcher": {
    "domains": ["example.com"]
  },
  "selectors": {
    "manga": {
      "title": "h1.title",
      "cover": "img.cover@src",
      "author": ".author"
    },
    "chapter": {
      "list": "ul.chapter-list li a",
      "title": ".chapter-title",
      "number": "regex:第(\\d+)话"
    },
    "currentUrl": {
      "isMangaPage": "regex:/manga/[a-z0-9-]+$",
      "isChapterPage": "regex:/manga/[a-z0-9-]+/c\\d+"
    }
  }
}

legado-v3 兼容（V1 主力路径）

用户从 legado 社区拿到漫画书源 JSON URL（或文件）→ 我们的设置页粘贴 → loader detect format: "legado-v3" → adapter 转换 → 写入内部模型。零成本起步：首次安装可一键导入"社区精选 legado 漫画书源包"，瞬间获得几百个源。

合规约束（Hard Rule）

不在仓库内硬编码任何盗版漫画站 URL。代码、测试 fixture、文档 — 一律使用 https://example.com/manga/... 占位。真实运行规则由用户运行时导入，不打包进 release artifact。legado 兼容代码本身可开源；默认订阅 URL 不指向真实盗版源。

茶色/深绿思源宋体 + 黑体

Preview 站

独立的静态预览页，用 Tooling Mono 默认风格实现，覆盖所有 4 类宿主（Sync Bar / Popup / Options / Mobile 四屏），带状态切换控件（normal / empty / loading / error），用于 v1 编码前的 UI 走查。

项说明
路径site/preview/index.html（与设计文档同仓库，CF Pages 自动一并部署）
访问部署后：https://manga-tracker.pages.dev/preview/
本地：cd site && python3 -m http.server 8000 → http://localhost:8000/preview/
状态切换顶部 toolbar 按钮组 + URL query（?state=empty、?state=loading、?state=error）
主题顶部 Dark / Light 切换，状态保存到 localStorage
Mock 数据12 部虚构作品（覆盖长 / 短标题、不同进度、完结 / 连载、4 个虚构源、不同更新时间）；7 条最近阅读
合规所有源标识 example-1 ~ example-4，URL example.com 占位；不指向任何真实漫画站
Preview 站只覆盖 UI 走查，不替代真实环境验证：扩展真实注入、WebView 真实抓取、源规则真实匹配仍需在真实漫画站测试（这部分 mock 不出价值）。

技术栈

层选型理由
Monorepopnpm workspaces + Turborepo单人维护成本最低；Turborepo 任务编排开箱即用
语言TypeScript 5.x（strict）跨端类型唯一来源依赖 TS 推导和共享
Lint/FormatBiome替代 ESLint+Prettier，monorepo 性能更好
测试Vitest统一 packages + apps，配置最简
扩展框架Plasmo 0.89+多浏览器（Chrome/Edge/Firefox/Safari），CSUI 内建支持 Sync Bar 注入；CRXJS 在跨浏览器场景下要自己拼 manifest
移动端容器Capacitor 6.xReact 代码与扩展共享率最高（vs RN/Flutter）
UI 库React 18 + Tailwind v4v4 用 Lightning CSS，扩展场景体积小
状态Zustand轻量，跨端 store 代码可部分共享
SchemaZod边界校验（用户导入数据 / 源规则）
本地存储（扩展）Dexie.js (IndexedDB)避免裸 IDB API
本地存储（mobile）@capacitor-community/sqlite带加密支持

Cloudflare 部署

本设计文档站（site/）部署到 Cloudflare Pages，从 GitHub 仓库自动构建。配置在 site/wrangler.toml 和 site/_headers。

一次性配置

  1. 把项目推到 GitHub： 
    cd /Users/falcom/Documents/projects/manga-tracker
git init -b main
git remote add origin git@github.com:YeomanYe/manga-tracker.git
git add .
git commit -m "chore(repo): initial design + scaffolding"
git push -u origin main
  2. 登录 Cloudflare Pages 控制台
  3. 点 Create a project → Connect to Git，选 YeomanYe/manga-tracker
  4. 构建配置：
    • Framework preset: None
    • Build command: 留空（纯静态 HTML，无需构建）
    • Build output directory: site
    • Root directory: /
  5. 点 Deploy。Pages 会给你一个 manga-tracker.pages.dev 域名，每次 push 到 main 自动部署

自定义域名（可选）

如果你要绑自己的域名（比如 docs.yourbrand.com），在 Pages 项目的 Custom domains 里添加，按提示加 CNAME 记录到你的 DNS（推荐用 Cloudflare DNS，全自动）。

预览部署（PR 自动）

Pages 默认会为每个 PR 自动起一个 preview 部署，URL 形如 <commit-sha>.manga-tracker.pages.dev。一个人开发时这个特性也很有用：可以在合并前先看新版长什么样。

本地预览

# 用任何静态服务器都行，例如：
cd site
python3 -m http.server 8000
# → http://localhost:8000

# 或者用 wrangler 模拟 Pages 环境：
npx wrangler pages dev site
